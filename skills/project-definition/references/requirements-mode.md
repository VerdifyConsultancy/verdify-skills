# Requirements quality

A requirement is necessary, unambiguous enough to test, solution-neutral where possible, and linked to evidence. Avoid combining multiple obligations in one ID.

Functional requirements describe observable capability. Non-functional requirements describe qualities or constraints such as security, performance, availability, auditability, portability, operability, and maintainability.

Acceptance criteria should name an observable state or behavior and the evidence expected. “Works correctly,” “user friendly,” and “production ready” are not sufficient.
